# 28_PROMPT_SPECIFICATION_PART3

Version: 4.0

Status: Production

------------------------------------------------------------------------

# Purpose

This document defines the final prompt contract used by the Founder
Intelligence Platform before report assembly.

The objective is to produce deterministic, auditable narrative that
integrates seamlessly with the report rendering pipeline.

------------------------------------------------------------------------

# Output Contract

The language model must return narrative that maps directly to the
report template.

The output must contain exactly the following sections:

1.  Executive Summary
2.  Current Position
3.  Evidence Assessment
4.  Benchmark Context
5.  Key Strengths
6.  Primary Risks
7.  Evidence Gaps
8.  Priority Actions
9.  Decision Readiness
10. 30-Day Roadmap
11. 90-Day Roadmap
12. Closing Assessment

No additional sections may be created.

------------------------------------------------------------------------

# Formatting Rules

The response must:

-   use Markdown;
-   use the supplied section headings;
-   use concise paragraphs;
-   use bullet lists only where appropriate;
-   avoid HTML;
-   avoid emojis;
-   avoid markdown tables unless explicitly requested by the
    application.

------------------------------------------------------------------------

# Narrative Rules

The model must:

-   explain deterministic results;
-   remain internally consistent;
-   reference supplied evidence;
-   acknowledge uncertainty where appropriate;
-   preserve the confidence level supplied by the application.

The model must not introduce new facts or reinterpret deterministic
outputs.

------------------------------------------------------------------------

# Failure Conditions

The response is considered invalid if it:

-   invents evidence;
-   changes deterministic scores;
-   changes recommendation order;
-   omits required sections;
-   contradicts supplied inputs;
-   guarantees business outcomes;
-   provides legal, tax or investment advice.

Invalid responses should be regenerated by the application using the
same structured inputs.

------------------------------------------------------------------------

# Versioning

Every generated report must include the following metadata supplied by
the application:

-   report_version
-   prompt_version
-   scoring_model_version
-   generated_timestamp

The language model must render these values exactly as supplied.

------------------------------------------------------------------------

End of Part 3.
